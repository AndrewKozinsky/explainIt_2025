# Перевод предложений из книг и фильмов

## Что делает функционал
Пользователь выделяет слово в предложении книги или субтитров и получает:
- перевод всего предложения;
- перевод фразы, в которую входит выделение;
- короткие примеры для этой фразы.

Цель — учить лексику в контексте конкретного предложения и материала.

## Как работает сервер

### GraphQL API
Клиент использует 2 query и 2 mutation:
- `translate_get_sentence_translation` — получить сохранённый перевод предложения по `sentenceId`;
- `translate_get_phrase_translation` — получить фразовый перевод по `sentenceId` и диапазону выделения;
- `translate_translate_sentence` — сгенерировать перевод предложения;
- `translate_translate_phrase` — сгенерировать фразовый перевод.

### Общая схема обработки
`translate.resolver.ts` передаёт запросы в CQRS handlers:
- `TranslateSentenceHandler`;
- `TranslatePhraseHandler`.

Для фразы сервер:
1. Проверяет корректность offset-диапазона.
2. Создаёт/резолвит `pending` запись в `SentencePhraseTranslation`.
3. Запрашивает генерацию у LLM.
4. Парсит ответ, резолвит итоговую фразу как подстроку предложения.
5. Обновляет запись в `ready` или `error`.

Важно: путь `translate_translate_phrase` ориентирован на генерацию (не на возврат существующего cached значения через query).

### Новый формат ответа фразового перевода
Для `translate_translate_phrase` сервер больше не использует JSON-промпт/парсинг.

LLM получает инструкцию вернуть plain text по строкам:
1. фраза на исходном языке;
2. перевод фразы;
3+. пары строк примеров: `пример` / `перевод примера`.

Парсер `parsePhraseTranslationResult`:
- берёт 1-ю и 2-ю строки как `phrase` и `translate`;
- оставшиеся строки читает парами и собирает `examples`;
- отбрасывает неполные пары.

### Хранение примеров в БД
В `sentencePhraseTranslation.repository.ts` примеры хранятся как плоский массив строк:
- формат в БД: `[text1, translate1, text2, translate2, ...]`;
- при чтении массив декодируется обратно в `[{ text, translate }]`.

Это уменьшает сложность сериализации и исключает legacy-ветки JSON-совместимости.

### Доступ и ограничения
- Доступ к чтению/созданию перевода проверяется через сервис доступа.
- Для некоторых режимов действует дневной лимит (учёт в Redis).
- При успешной генерации списываются токены/баланс согласно режиму доступа.

## Как работает клиент

Основная логика находится в `face/_pages/media/detailsBlock/DetailsBlock/fn/getAnalysis.ts`.

### Смена предложения
Когда выбран новый `sentenceId`, клиент делает три действия параллельно:
1. получает или создаёт перевод предложения;
2. запрашивает все готовые фразовые переводы предложения;
3. получает/создаёт перевод для текущего выделенного слова.

После этого:
- объединяет результаты в локальный кэш фраз (по диапазонам offset);
- обновляет store (`sentenceTranslation`, `wordAnalysis`, `wordAnalyses`).

### Смена слова внутри того же предложения
Если `sentenceId` тот же, но поменялось выделение:
1. клиент сначала ищет подходящий фразовый перевод в локальном кэше по диапазону;
2. если найдено — показывает мгновенно, без дополнительного server round-trip;
3. если не найдено — вызывает `translate_translate_phrase`, добавляет результат в кэш и обновляет UI.

Для выбора подходящей фразы используется правило «самая короткая фраза, содержащая выделенный диапазон».

### Клиентский state
`detailsStore` сейчас хранит только результатный state для UI:
- `sentenceTranslation`;
- `wordAnalysis`;
- `wordAnalyses`.

Технический кэш фразовых переводов (с offset-диапазонами) вынесен из store в локальный `ref` внутри `useGetAnalysis`, чтобы не раздувать глобальный state и не вызывать лишние перерисовки.
